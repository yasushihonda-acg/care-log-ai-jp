1 アーキテクチャ設計

概要

プロジェクト名: AI介護記録アプリ (care-log-ai-jp)

音声または自然言語テキストで入力された介護記録をGoogle Gemini (Flashモデル) で解析・構造化し、Vercel Postgresに保存・可視化するWebアプリケーションです。

2 データベース設計

テーブル定義 (care_records)

CREATE TABLE care_records (
  id SERIAL PRIMARY KEY,
  record_type VARCHAR(50) NOT NULL, -- 'meal', 'vital', 'excretion', etc.
  details JSONB NOT NULL,           -- 柔軟な構造化データ
  recorded_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

JSONBデータ構造パターン

AI解析の成功率を高めるため、詳細は基本的にString型として保存されます。

食事 (meal)

{
  "main_dish": "全粥",
  "amount_percent": "80",
  "fluid_ml": "200",
  "fluid_type": "お茶"
}

バイタル (vital)

{
  "temperature": "36.8",
  "systolic_bp": "124",
  "pulse": "72"
}

3 API 仕様書

POST /api/parse

自然言語テキストを解析し、構造化データへ変換します。フロントエンドのフィールド設定を受け取り、動的にプロンプトを構築します。

Request: 
{
  "text": "昼食は全粥8割、お茶200mlでした。",
  "fieldSettings": { ... }
}

GET /api/records

記録一覧を取得します（最新100件）。

POST /api/chat

RAG (Retrieval-Augmented Generation) を利用したAI相談機能。直近50件の記録をコンテキストとして回答します。

4 機能仕様詳細

AI解析ロジック (ハイブリッド設計)

設計思想: Loose Schema & Frontend Validation

AIに対して厳密な数値変換（例: "8割"→80）を強制すると、パースエラーが多発します。
本システムでは、AIには「文字列としてそのまま抽出すること」を指示し、型制約を緩めることで入力成功率を最大化しています。

プロンプトエンジニアリング: 複合情報の分離

Gemini Flashモデルに対し、具体的なFew-Shot（例示）を与えることで、情報の分離精度を高めています。

  • 水分摂取の分離: 「お茶200ml」という入力に対し、fluid_type: "お茶" と fluid_ml: "200ml" に分離して記録するよう指示しています。

動的フィールド設定

施設ごとに異なる記録項目に対応するため、localStorage を利用したマスター設定機能を実装しています。

  • 自動キー生成: ユーザーは日本語ラベルを入力するだけで、システム用ID (f_xxxx) が自動生成されます。
  • AI連携: 現在の設定項目が解析リクエスト時にAIへ送られ、AIは未知のカスタム項目であってもラベル名から推測して値を割り当てます。
